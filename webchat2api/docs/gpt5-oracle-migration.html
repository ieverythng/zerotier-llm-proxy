The Goal

Simulate tool calling and agentic behavior through prompt engineering + a tool execution layer

🎯 Target Architecture

Build a tool simulation layer between Hermes and webchat2api that transforms text-only completions into a functional agentic loop. The core insight: ChatGPT's "tool calling" in the UI is just structured JSON — we can replicate it through prompt engineering.

Core Techniques

Five techniques to unlock agentic behavior through a text-only proxy

1. Structured JSON Output

Force ChatGPT to produce machine-readable JSON through strict system prompts and schema enforcement.

System-Level Instructions

{
  "model": "gpt-5.5-thinking",
  "temperature": 0.1,
  "messages": [
    {"role": "system", "content": "You ALWAYS respond in strict JSON. No explanations outside JSON. Follow this schema: {\"action\": string, \"parameters\": object}"},
    {"role": "user", "content": "Search for weather in Tokyo"}
  ],
  "stop": ["}"]
}

Schema Enforcement

  • Include example output in the prompt to ground the model
  • Use low temperature (0.0–0.2) for deterministic outputs
  • Set stop sequences at } to truncate extra commentary
  • Post-process with JSON validator — re-prompt on failure

2. Tool Calling Simulation

Describe tools in the system prompt, parse JSON responses, execute locally, feed results back.

Tool Registry Pattern

System: You can call these tools:
1. web_search(query: str) → returns search results
2. read_file(path: str) → returns file content  
3. terminal(command: str) → runs shell command
4. calculator(expression: str) → returns numeric result

Format tool calls as JSON:
{"tool": "tool_name", "args": {"arg1": value}}

User: What's the disk usage on /home?

Expected response:
{"tool": "terminal", "args": {"command": "df -h /home"}}

Execution Loop

  1. Hermes sends query + tool registry to Tool Layer
  2. Tool Layer wraps as prompt, sends to webchat2api
  3. Model returns JSON tool call in text response
  4. Parser extracts tool name and arguments
  5. Executor runs the tool locally
  6. Result fed back as new message: {"role": "user", "content": "Tool result: ..."}
  7. Model continues reasoning or calls another tool

3. Agent Mode Simulation

ChatGPT's "Agent Mode" and "Developer Mode" are prompt-level behaviors — replicable through the proxy.

Conversation Style

Set conversation_style=precise (if supported by webchat2api) for structured outputs. Use gpt-5.5-thinking for multi-step reasoning tasks.

Developer Mode Prompt

You are in developer mode: you can reason step-by-step, 
decide which tools to call, and output structured JSON only.

Plan your approach before acting. Use a "reasoning" field 
to describe your logic before the tool call.

Multi-Step Reasoning

Maintain conversation state across rounds. Each round: query → tool call → execute → feedback → next decision. This mimics ChatGPT's multi-step agentic behavior.

5. Attack Surface Analysis

What webchat2api exposes and what ChatGPT's UI does that we can replicate.

webchat2api Endpoints

  • POST /v1/chat/completions — OpenAI-compatible completions endpoint
  • GET /openapi.json — Full API schema documentation
  • GET /docs — Swagger UI for interactive testing

What ChatGPT's UI Does (Replicable)

  • System prompts: Injected server-side — we can set our own via the messages array
  • Tool descriptions: Sent as structured data — we replicate in system prompt text
  • Multi-turn conversations: Maintained as message history — we manage the buffer
  • Plugin calls: JSON objects with tool keys — we parse and execute locally

What We Cannot Replicate

  • Built-in tools Browse, Code Interpreter, DALL-E are UI-only — not accessible via proxy
  • MCP servers Developer Mode MCP integration is human-driven ChatGPT feature
  • Real-time streaming webchat2api supports it but parsing mid-stream is unreliable

Migration Roadmap

Phased approach from prompt engineering to full agentic loop

Phase 1 — Structured Output Foundation

Week 1–2
  • Implement strict JSON system prompts for GPT-Oracle queries
  • Add JSON validation + retry logic to ask-gpt5.sh
  • Test with low temperature (0.1) and stop sequences
  • Verify reliability: 95%+ successful JSON parse rate

Phase 2 — Tool Simulation Layer

Week 3–4
  • Build tool registry: define available tools with schemas
  • Create prompt template that injects tool descriptions into system message
  • Implement JSON parser to extract tool calls from responses
  • Build executor that maps tool names to local Hermes tools

Phase 3 — Agentic Feedback Loop

Week 5–6
  • Implement conversation state management (message buffer)
  • Build feedback loop: tool result → new message → model continues
  • Add max iteration limit to prevent infinite loops
  • Test multi-step workflows: search → read file → analyze → summarize

Phase 4 — Integration & Hardening

Week 7–8
  • Integrate tool layer into gpt5-consultant skill
  • Add error handling: invalid JSON → re-prompt, tool failure → fallback
  • Implement rate limit tracking and quota management
  • Performance testing: latency, success rate, token efficiency

Phase 5 — Production & Monitoring

Week 9+
  • Deploy tool layer as Hermes plugin or standalone service
  • Add metrics: success rate, avg iterations per query, quota usage
  • Set up alerts for proxy breaking changes (ChatGPT updates)
  • Document workflows and create example recipes
