Optional Context Layer

Headroom + Hermes Integration

Operational guide for compressing Hermes tool output and session history before it reaches the local llama.cpp backend.

Purpose

Headroom targets prompt growth, not GPU decode speed. It sits between Hermes and LiteLLM, compressing large logs, JSON, tool outputs, and stale context while preserving originals for retrieval.

The prior stall was caused by a 54,737-token Hermes tool-agent prompt, not a 0.15 tok/s model. Headroom is intended to reduce the input size that produces queueing and slow long-context decode.

Architecture

Hermes Agent in WSL
  -> Headroom proxy :8787 on Windows host
  -> LiteLLM :4000 on Windows host
  -> llama.cpp :8080 on Windows
  -> GPU

Headroom runs on Windows. Hermes reaches it over the existing Windows host bridge; LiteLLM and llama.cpp retain the existing API and ZeroTier layout.

Scripts

ScriptPurpose

Install And Test

# Install only. It does not change Hermes routing.
.\scripts\windows\Install-HeadroomHermes.ps1

# Start only. Hermes continues to call LiteLLM directly.
.\scripts\windows\Start-HeadroomHermes.ps1
.\scripts\windows\Get-HeadroomHermesStatus.ps1

The proxy starts with telemetry disabled, a 1,000-token compression minimum, protection for the most recent 12 turns, and exclusions for read_file and headroom_retrieve.

Controlled Cutover

# Start the normal local stack and Headroom, then route Hermes through it.
.\scripts\windows\Start-Qwen36HeadroomStack.ps1 -RouteHermesThroughHeadroom

# Or route an already-running healthy proxy.
.\scripts\windows\Set-HermesHeadroomRoute.ps1 -Enable

# Roll back immediately to LiteLLM.
.\scripts\windows\Set-HermesHeadroomRoute.ps1 -Disable
Do not cut over production Discord traffic until a fresh thread passes tool-call fidelity tests. Compression must reduce prompt tokens without making the agent re-run commands or lose required file content.

Runtime Changes

  • Hermes still uses qwen36-turbo-hermes and OpenAI chat completions.
  • When enabled, only the provider URL changes from LiteLLM to Windows Headroom.
  • Headroom forwards to http://127.0.0.1:4000/v1.
  • Headroom keeps a local CCR store. The optional Hermes retrieval plugin lets the agent request a compressed original on demand.
  • The existing ZeroTier endpoint remains LiteLLM on port 4000; remote Codex clients do not need a new endpoint.

Validation Criteria

  1. Run a fresh Discord thread with a representative multi-tool task.
  2. Compare Headroom token statistics to llama.cpp prompt token timings.
  3. Verify tool calls complete correctly and the agent does not repeat work.
  4. Check first-token latency and decode throughput separately.
  5. Use the route toggle to roll back if correctness or latency regresses.

Current Trial

The Windows proxy is installed with tool-result interception, Hermes routes through it, and the headroom_retrieve plugin is enabled. A real Hermes request reached the proxy with 20,181 input tokens. It did not invoke a tool, so there was no tool result to compress and Headroom correctly recorded zero savings.

Use a real multi-tool Discord task to complete the evaluation. Inspect http://127.0.0.1:8787/stats after the task and compare prompt tokens, first-token latency, decode speed, and tool correctness. The route script retains a timestamped backup for immediate rollback.

DevSpace Boundary

DevSpace is useful for connecting ChatGPT to local workspaces through MCP. It is not an OpenAI-compatible model service and cannot be used as Hermes' Oracle backend. Keep Oracle work separate from the Headroom proxy evaluation.
