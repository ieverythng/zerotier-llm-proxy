How the Windows Headroom proxy reduces long Hermes tool history before it reaches LiteLLM and llama.cpp.

Hermes keeps the same model name and OpenAI chat-completions transport. The only provider change is the proxy URL. Remote ZeroTier clients continue to use LiteLLM on port 4000.

What Headroom Manages

LayerBehaviorWhy it matters here
Tool-result interceptorExamines terminal, file, search, and session outputs before the following model call.Large agent outputs are the source of the previous 54k-plus prompt stalls.
Content routerClassifies structured data, prose, code, and repeated content, then selects a compression strategy.Only eligible stale content is rewritten; current user instructions remain verbatim.
CCR retrieval storeStores originals locally and emits retrieval markers when content is compressed.Hermes can call headroom_retrieve instead of re-running a command.
Compression cacheReuses prior transformations for unchanged content.After the first pass, observed compression overhead was roughly 39-130 ms.
Prefix preservationLeaves stable instructions and recent context intact.Preserves llama.cpp KV-cache behavior and protects current task state.

Operational Controls

# Inspect accumulated live savings
Invoke-RestMethod http://127.0.0.1:8787/stats

# Immediate rollback to the original LiteLLM route
.\scripts\windows\Set-HermesHeadroomRoute.ps1 -Disable
Do not treat proxy savings as model decode speed. Headroom reduces prefill/context work; llama.cpp decode tok/s remains the model-server metric.
