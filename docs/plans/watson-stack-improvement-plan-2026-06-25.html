Executive decision

Highest ROI first: instrument everything, then run a context/quantization/tool-use sweep. The current llama.cpp TurboQuant/MTP backend is already strong. The biggest gains now are likely from reducing wasteful prompt growth, protecting VRAM, better routing, and knowing exactly when Watson is doing model work versus tools versus delegation.

1. Throughput improvement framework

Separate model speed from agent speed. A fast decode number is not enough if the agent repeats tools, produces malformed JSON, or spends 80% of wall time in prefill.

Metric familyWhat to record
ModelPrompt tok/s, decode tok/s, TTFT, end-to-end wall time, stop reason.
ContextInput tokens, generated tokens, context remaining, Headroom savings, retrieval calls.
ToolsTool-call validity, repeated tool calls, terminal latency, file write success, task completion.
SystemVRAM peak, RAM peak, GPU util, process state, queue depth, port health.

Context ceiling strategy

Do not jump straight to 100k

Use a monotonic sweep: 65k baseline, 80k, 96k, 112k, 128k. Stop at the first mode that fails health, tool use, or latency.

Test powers and practical plateaus

64k and 128k are good anchor points, but 80k and 96k may be the real sweet spots on 16GB VRAM.

Measure usable context, not just loadability

A model that loads at 128k but takes too long to prefill or loses tool fidelity is not a production win.

Suggested sweep order: 1. Q3 MTP, current KV profile, 65k baseline 2. Q3 MTP, 80k, 96k, 112k, 128k 3. Q3 MTP, lower KV cache precision variants 4. Q2/Q2_K model, same context ladder 5. Headroom on/off across each successful context profile 6. Tool benchmark gate before production routing

vLLM evaluation

vLLM is worth testing, but not because GGUF magically becomes faster there. The value is server behavior: continuous batching, prefix caching, Prometheus/Grafana observability, tool-calling support, and FP8 KV cache experiments.

Recommendation: run vLLM as a branch with a native HF/AWQ/GPTQ/FP8 model or a smaller worker model first. Treat GGUF-on-vLLM as experimental until it beats llama.cpp on your own benchmark.

Hermes dashboard over ZeroTier

Telemetry backend

FastAPI/Starlette service on Windows or WSL. It polls nvidia-smi, Windows process state, LiteLLM, Headroom, llama.cpp logs, and Hermes gateway state.

Live UI

Single-page dashboard with WebSocket/SSE updates: GPU, VRAM, RAM, active model, active agent, queue, context, tool state, and route state.

True typing state

Replace Discord’s coarse “Watson is typing” with internal states: idle, prefill, decode, tool call, delegate task, memory update, waiting.

Voice channel plan

Do voice as a sidecar, not inside the main LLM process. Start with a low-VRAM route: Discord voice capture → VAD → ASR → Hermes text turn → TTS → Discord voice playback.

LayerPreferred first passWhy

Premium TTSDia / Chatterbox / OrpheusUse only in “voice mode” when VRAM is available.

Swarm and fixed VRAM

On 16GB VRAM, the swarm should be scheduled, not always resident. The realistic design is a VRAM-aware job router with modes:

Mode normal: Qwen27B + Headroom + dashboard Mode voice-lite: Qwen27B + CPU TTS Mode high-context: Qwen27B lower KV / lower quant Mode swarm-small: unload Qwen27B, run 3B/4B workers Mode 14B-worker: unload Qwen27B or reduce context heavily Mode iTrader: reserve VRAM budget, queue noncritical jobs

Global Lazarus controller

Health chain

Port check → /models check → synthetic chat → Hermes route check → Discord heartbeat → tool smoke test → VRAM budget check.

Recovery policy

Restart only the failing layer first. Kill/reload GPU model only when llama.cpp is unhealthy or VRAM fragmentation makes recovery impossible.

Mode lock

A single state file prevents two scripts from trying to load different GPU modes at once.

watson-lazarus.yaml services: llama: { port: 8080, exclusive_gpu: true, health: /v1/models } litellm: { port: 4000, depends: [llama], health: /v1/models } headroom: { port: 8787, depends: [litellm], health: /health } hermes: { wsl: true, health: "hermes gateway status" } discord: { wsl: true, health: "bot heartbeat" } policies: cooldown_seconds: 30 max_restarts_per_hour: 5 gpu_guard_mib: 768

Implementation roadmap

Phase 0
Freeze baseline. Save exact current llama.cpp build, model path, context, KV types, LiteLLM config, Hermes route, and one representative Discord transcript.
Phase 1
Benchmark harness. Add repeatable tests for throughput, context, Headroom, tool calls, and Discord end-to-end completion.
Phase 2
Context sweep. Test 80k/96k/112k/128k with current Q3 before trying Q2. Accept only if tool fidelity and latency remain usable.
Phase 3
Dashboard. Ship a ZeroTier dashboard with GPU, VRAM, RAM, context and true agent state. This becomes your main cockpit.
Phase 4
Voice-lite. Add Discord VC with CPU-friendly TTS first. Promote higher-quality TTS models only under a VRAM scheduler.
Phase 5
Lazarus + scheduler. Use mode-based recovery and VRAM locks before adding iTrader and worker swarms.
