Source-backed levers

llama.cpp exposes prompt-cache and timing signals that are directly relevant here: cache_n, prompt_n, prompt_ms, prompt_per_second, predicted_ms, and predicted_per_second.

The server knobs to test first are n_ctx, n_batch, n_ubatch, Flash Attention, KV K/V cache type, --cache-prompt, --cache-reuse, --metrics, and --perf.

Prefix-cache model

vLLM's automatic prefix caching documentation describes the same workload class: repeated long-prefix and multi-round chat workloads get faster during prefilling, while decode-heavy workloads benefit less.

That maps directly onto Hermes/Discord: long channel history and repeated tool output can be either a prefill tax or a cache/compression opportunity.

Stage 1
Upgrade measurement. Add streaming TTFT, timings parsing, cache labels, controlled cache-busting, and GPU snapshots.

Runtime sweeps. Test batch/ubatch, KV cache variants, Headroom route, then 80k and 96k candidates using the winning 65k settings.
Stage 4
Agentic workflow score. Rank short chat, long continuation, multi-tool Discord task, file task, tool JSON, Lazarus recovery, and VRAM headroom.

Deslop guardrails for code work

Future harness changes should be narrow and behavior-preserving except for explicit metrics additions.

  • Do not rewrite the harness broadly.
  • Extract repeated HTTP timing and cache parsing into helpers.
  • Keep launcher orchestration separate from benchmark measurement.
  • Fail explicitly when metrics are unavailable.
