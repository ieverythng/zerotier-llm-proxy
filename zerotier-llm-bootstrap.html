ZeroTier LLM Bootstrap

Bootstrap Doc v2.0

Single-source implementation guide for cross-device LLM serving via ZeroTier. Pass this file to any agent on any machine — it contains everything needed.

Quick Start (Copy-Paste)

On Windows PC (Inference Host):

# 1. Install LiteLLM proxy
pip install 'litellm[proxy]'

# 2. Create config directory
mkdir %USERPROFILE%\.litellm

# 3. Save litellm-config.yaml (see Config Files section below) to:
#    C:\Users\Juan\.litellm\config.yaml

# 4. Ensure llama.cpp is running on port 8080

# 5. Start LiteLLM proxy
$env:OPENAI_API_KEY="***"
litellm --config "$env:USERPROFILE\.litellm\config.yaml" --port 4000 --host 0.0.0.0

# Use the install script from this repo:
cd /path/to/zerotier-llm-proxy
./scripts/unix/install-codex-client-config.sh

# Test it (requires --profile flag):

Key Design Decisions

Why LiteLLM on Windows?

LiteLLM is a proxy layer, not an inference engine. It sits between clients and the model server, translating API formats and managing routing. By placing it on the Windows PC alongside llama.cpp:

  • Single configuration point — configure once, all devices connect
  • ZeroTier binding — LiteLLM binds to 0.0.0.0, exposing port 4000 on the ZeroTier interface
  • Responses API translation — Codex CLI/App uses OpenAI's Responses API; LiteLLM translates it to Chat Completions that llama.cpp understands
  • No duplicate installs — Mac and Linux only need Codex CLI/App, no proxy software

Why llama.cpp stays on localhost?

llama.cpp binds to 127.0.0.1:8080 — it is the raw inference engine and does not need external exposure. LiteLLM handles all network-facing concerns:

  • Authentication and rate limiting
  • API format translation (Responses to Chat Completions)
  • Request queuing when multiple clients connect simultaneously

Codex Provider Architecture

Important: Codex routes models to providers inside its compiled Rust binary. There is no per-model provider override in config or model catalogs. The only way to route a specific model through a custom provider is via the --profile flag, which sets model_provider.

Provider Registration (all clients)

Add this to ~/.codex/config.toml:

[model_providers.qwen36-zerotier]
name = "qwen36 via Windows ZeroTier LiteLLM"
base_url = "http://10.88.140.94:4000/v1"
wire_api = "responses"

This does not change the global default model or override the OpenAI provider. It only registers a named provider that profiles can reference.

Selectable Profile (CLI use)

Create ~/.codex/qwen36-zerotier.config.toml:

model = "qwen36-turbo-hermes"
model_provider = "qwen36-zerotier"
model_context_window = 65536
model_max_output_tokens = 8192

Then use it with:

codex exec --profile qwen36-zerotier "Say hello"

Model Catalog (App model list)

A merged catalog at ~/.codex/model-catalogs/qwen36-plus-bundled.json adds qwen36-turbo-hermes to the model selector dropdown. The install script generates this automatically from codex debug models --bundled.

Codex Desktop App Integration

The Problem

When you select qwen36-turbo-hermes from the Codex Desktop app model dropdown, the app spawns a CLI session with -m qwen36-turbo-hermes but does not pass --profile. Without the profile, Codex routes the model through the default OpenAI provider, which fails because that model does not exist on OpenAI's servers.

The Solution: Shell Wrapper

A shell script replaces the codex binary in PATH. It intercepts every invocation, detects when -m qwen36-turbo-hermes is present, and auto-injects -p qwen36-zerotier before delegating to the real Codex binary.

Setup Steps

  1. Rename the original codex script: 
    # The npm-installed codex is at ~/.local/bin/codex (or wherever npm put it)
mv ~/.local/bin/codex ~/.local/bin/codex.real
  2. Create the wrapper at ~/.local/bin/codex: 
    #!/usr/bin/env bash
# Wrapper: auto-inject qwen36-zerotier profile for qwen36-turbo-hermes model.
REAL_CODEX="/home/juanbeck/.local/bin/codex.real"

ARGS=("$@")
NEED_PROFILE=0

for i in "${!ARGS[@]}"; do
    if [[ "${ARGS[$i]}" == "-m" || "${ARGS[$i]}" == "--model" ]]; then
        next=$((i + 1))
        if [[ $next -lt ${#ARGS[@]} ]]; then
            model="${ARGS[$next]}"
            if [[ "$model" == "qwen36-turbo-hermes" ]]; then
                NEED_PROFILE=1
            fi
        fi
    fi
    if [[ "${ARGS[$i]}" == -m=* ]]; then
        model="${ARGS[$i]#-m=}"
        [[ "$model" == "qwen36-turbo-hermes" ]] && NEED_PROFILE=1
    fi
    if [[ "${ARGS[$i]}" == --model=* ]]; then
        model="${ARGS[$i]#--model=}"
        [[ "$model" == "qwen36-turbo-hermes" ]] && NEED_PROFILE=1
    fi
done

if [[ $NEED_PROFILE -eq 1 ]]; then
    has_profile=0
    for arg in "${ARGS[@]}"; do
        [[ "$arg" == "-p" || "$arg" == "--profile" || "$arg" == -p=* || "$arg" == --profile=* ]] && { has_profile=1; break; }
    done
    [[ $has_profile -eq 0 ]] && ARGS=("-p" "qwen36-zerotier" "${ARGS[@]}")
fi

exec "$REAL_CODEX" "${ARGS[@]}"
    chmod +x ~/.local/bin/codex
  3. Verify: 
    # Should route through qwen36-zerotier automatically:
codex exec -m qwen36-turbo-hermes "Say hello"

# Should still use default OpenAI provider (unaffected):
codex exec -m gpt-5.5 "Say hello"

How the Desktop App Uses It

The Codex Desktop app's start.sh resolves CODEX_CLI_PATH via command -v codex, which returns ~/.local/bin/codex — our wrapper. Every session the app spawns goes through the wrapper, so selecting qwen36-turbo-hermes from the dropdown just works.

npm Update Resilience

Running npm update -g @openai/codex will overwrite the wrapper. After any npm update, restore it:

# Option A: Run the restore script (stored at ~/.codex/restore-codex-wrapper.sh)
~/.codex/restore-codex-wrapper.sh

# Option B: Manual restore
mv ~/.local/bin/codex ~/.local/bin/codex.real
# Re-create the wrapper script (see above)

Session Management (How Concurrent Clients Work)

Each HTTP Request is Independent

llama.cpp serves statelessly. Each /v1/chat/completions or /v1/responses request is a standalone operation:

  1. Codex on Mac sends a request with its own messages array — llama.cpp processes it — returns response — KV cache freed
  2. Codex on Linux sends a request with its own messages array — llama.cpp processes it — returns response — KV cache freed

No cross-contamination between clients. Each request carries its complete conversation history in the messages payload. The model does not "remember" between HTTP calls unless you explicitly include prior messages.

Where Context Lives

The conversation context is maintained by the client, not the server:

  • Codex App/CLI maintains its own conversation history locally. Each request sends the full message history to the model.
  • Hermes Agent maintains its own conversation history. Each delegation or tool call sends its own messages array.
  • llama.cpp just receives a batch of messages and generates a response — it has no persistent session state.

This means Mac Codex session A and Linux Codex session B are completely isolated. They share the model weights but have entirely separate conversation contexts.

KV Cache Behavior

The KV cache is the buffer that holds attention keys/values during inference:

  • Allocated once at startup (controlled by --ctx-size / n_ctx)
  • Used during a single request, then freed after response completes
  • Reused for the next request — no accumulation across clients

The real constraint: If a single conversation gets very long (e.g., 60K+ tokens), it fills the KV cache buffer. But this is per-request, not cumulative across clients.

Summary: Multiple clients = no OOM risk from context stacking. Each client's conversation is self-contained in its own messages array. The model weights are shared; the KV cache is reused per-request. Think of llama.cpp as a stateless function: f(messages) -> response.

Known Issues and Mitigations

Chat Completions Auth Quirk: LiteLLM proxy sometimes fails to pass credentials through to llama.cpp for /v1/chat/completions. The /v1/responses endpoint works cleanly. Mitigation: Set OPENAI_API_KEY as an environment variable when starting the proxy process (see Step 4 above).
Concurrent Request Latency: llama.cpp processes requests sequentially. When Hermes and Codex both send requests, they queue. Expect 10-30s latency per request for a 26B model. Do not run heavy tasks simultaneously.
Windows Firewall: If clients cannot reach port 4000 over ZeroTier, Windows Firewall may be blocking it. Add an inbound rule for TCP 4000 (see Step 6 above).
npm update breaks the wrapper: Running npm update -g @openai/codex overwrites ~/.local/bin/codex, replacing the shell wrapper with the original npm script. Restore with ~/.codex/restore-codex-wrapper.sh.

Token Budget Notes

Context Window: Your model has n_ctx: 65536 tokens. Codex sessions can be long — monitor context usage. Ensure llama.cpp is started with adequate --ctx-size (at least 32768, ideally 65536).

Troubleshooting

"Connection refused" from clients

  • Verify LiteLLM is running: check Windows terminal for Uvicorn running on http://0.0.0.0:4000
  • Check Windows Firewall: Get-NetFirewallRule | Where-Object {$_.LocalPorts -eq '4000'}
  • Test from Windows itself: curl http://127.0.0.1:4000/v1/models

"Missing credentials" error

  • Ensure OPENAI_API_KEY env var is set when starting LiteLLM
  • The value can be anything (e.g., ***) — llama.cpp does not validate it

Codex websocket errors

  • If you see websocket warnings, Codex is NOT using the LiteLLM proxy
  • Verify ~/.codex/config.toml has the [model_providers.qwen36-zerotier] section
  • LiteLLM's /v1/responses endpoint handles this cleanly — no websocket needed

Desktop app routes to OpenAI instead of qwen36-zerotier

  • The shell wrapper is missing or was overwritten by npm update
  • Restore: ~/.codex/restore-codex-wrapper.sh
  • Verify: head -1 ~/.local/bin/codex should show the wrapper shebang, not the npm script
  • Restart Codex Desktop after restoring the wrapper

llama.cpp not responding through proxy

  • Verify llama.cpp is running: curl http://127.0.0.1:8080/v1/models
  • Check LiteLLM config has correct api_base: http://127.0.0.1:8080/v1

ZeroTier routing wrong interface

  • If ip route get 10.88.140.94 shows your Wi-Fi gateway instead of a ZeroTier device:
  • Run sudo zerotier-cli set 3b19b3a716937e29 allowManaged=1
  • Reconnect ZeroTier and verify again
